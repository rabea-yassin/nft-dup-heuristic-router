# hsvHash — global HSV color histogram

A direct, **bit-exact** port of Buchner's `imagehash.colorhash(binbits=3)`.

> The paper's Section II-C(v) *prose* describes a block/region-based
> statistical hash (citing Tang et al. 2013), but the paper's actual
> numbers were generated by `imagehash`, whose only HSV-based function is
> `colorhash()` — a **global** histogram with no spatial blocks at all.
> That's what this ports (README.md's toolchain section documents why).

## Algorithm

A 42-bit fingerprint over quantized global color fractions:

1. Per pixel, compute intensity (PIL RGB→L) and hue/saturation (PIL
   RGB→HSV, 0–255-scaled — **not** 0–360° hue).
2. Classify: near-black (`intensity < 32`), near-gray (`s < 85`, not
   black), or colored. Split colored pixels by saturation into "faint"
   (`s < 170`) and "bright" (`s > 170`); `s == 170` falls in neither.
3. Build 6-bin hue histograms for faint and bright colored pixels.
4. 14 values total — `[frac_black, frac_gray, faint_bin0..5,
   bright_bin0..5]` — each quantized to `[0,7]`, then expanded to 3 bits
   via `colorhash()`'s own **non-monotonic** table (4/6 alias, 5/7 alias —
   replicated as-is, verified from source). 14 × 3 = 42 bits.

## Output

`hsvhash_t` = `uint64_t` (top 22 bits always zero). Compared by Hamming
distance (`hsvhash_distance`).

## Files

| File | Role |
|------|------|
| `hsvhash.h` / `hsvhash.c` | implementation (inlines its own PIL RGB→L / RGB→HSV; decodes via stb directly, so it does **not** use `common/`) |
| `hsvhash_sanity_check.c` | build-and-eyeball check vs `data/example/generated/` |

## Parity

**Bit-exact** — validated against live `imagehash.colorhash()` across 120
cases (80 synthetic boundary-targeting images + all 40 example fixtures),
zero mismatches on the raw 42-bit value.

## Notes — the float/double literal trap

PIL's RGB→HSV (`Convert.c` `rgb2hsv_row`) declares its locals as `float`
but uses **unsuffixed (double) literals** (`2.0`, `6.0`, `255.0`), so most
of the arithmetic runs at double precision and rounds down to float only
at each assignment. A naive all-`float` port is wrong on ~0.4% of pixels
at rounding boundaries; replicating that assignment-by-assignment rounding
(and `fmod`, not `fmodf`) is what makes it bit-exact. The same lesson
recurs in [`shash`](../shash)'s Gaussian-radius math.

Good for color-preserving edits (flips, rotations, overlays); blind to
color/background changes.
